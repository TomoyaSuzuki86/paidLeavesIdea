# # 目的


    このリポジトリでは、既存モック `paid_leave_ideas_mock.jsx` をたたき台にして、
    **「有給取得促進アイデアを閲覧し、企画担当が制度設計のヒントを得られる正式なWebサイト」**
    を実装する。

    このサイトは以下を満たすこと。

    - 単なるモックではなく、**本番品質の静的Webサイト**として完成させる
    - Codex CLI で安全に実装・修正・デプロイできる構成にする
    - Firebase Hosting にデプロイできる SPA 構成にする
    - 「条件選択して自動提案するツール」ではなく、**閲覧しながら発想が広がる編集的なサイト**にする
    - 普通の業務資料ではなく、**Apple系の明るく静かな洗練**を持つが、情報としては実務に耐えるものにする

    ## 実装方針

    ### 技術スタック
    - React
    - Vite
    - TypeScript を優先（既存 JSX は TSX へ移行可）
    - Tailwind CSS を使用してよい
    - Framer Motion は必要最小限のみ使用
    - Firebase Hosting にデプロイ可能な構成
    - バックエンド不要。静的データで成立させる

    ### フォント
    Google Fonts を利用し、以下の方向で実装すること。

    - 英数字・見出し: `Manrope`
    - 日本語本文: `Noto Sans JP`

    要件:
    - 読みやすさ最優先
    - Apple 風の余白と静けさに合うこと
    - 既存モックより洗練されていること
    - フォント読み込みは `index.html` または適切な方法で整理すること

    ### デザイン原則
    - 明るい背景
    - 余白を大きく取る
    - カードは白ベース + 淡い境界線 + ほんの少しのガラス感
    - 影は浅く、輪郭は柔らかく
    - 青系アクセントは控えめに使用
    - 情報密度はあるが、見た目は静かに保つ
    - Apple っぽさを意識するが、模倣ではなく「日本語の企画資料として自然」な見え方にする
    - 紫グラデーション、過剰なネオン、過剰アニメーションは禁止
    - スクロールで疲れないことを優先

    ## このサイトの役割

    このサイトは以下のような使われ方を想定する。

    - 人事企画担当が「有給取得促進施策のネタを探す」
    - 管理職施策、低取得者向け施策、推奨日施策などを比較する
    - 社内会議で「この切り口は使える」と話題にする
    - 詳細ページを見て、そのまま制度案・運用案のたたき台にできる

    **重要:**  
    このサイトは「制度診断ツール」ではない。
    ユーザーに条件を入力させて自動提案する方向にはしない。
    あくまで「閲覧してアイデアを得る」「比較しながら制度化のヒントを得る」サイトにする。

    ## アイデアの深掘り要件

    ### 既存12案はすべて詳細化すること
    既存の 12 アイデアは、短い summary のまま使わない。
    すべて詳細化して、制度設計のヒントとして使える粒度にする。

    ### さらに追加アイデアを最低8件足すこと
    既存12件に加えて、**最低8件** 追加すること。
    合計 **20件以上** のアイデアライブラリにする。

    追加アイデアの条件:
    - 低コスト寄り
    - 実務でありそうで、少しひねりがある
    - 既存12件と意味が重複しない
    - 「推奨日」「管理職」「低取得者」「属人化」「連絡」「半休」「文化形成」などの軸を広げる
    - 抽象的な精神論ではなく、制度 / 運用 / 導線 / 文言 / 会議設計まで落ちること

    ### 追加候補の方向性例
    以下はそのまま使ってもよいし、より良い案に置き換えてもよい。

    - 休暇予定者の会議を自動で候補除外するルール
    - 月初に「今月休む人」を宣言しやすいミニ共有フォーマット
    - 休んだ人本人ではなく、代理対応者の負荷を見える化する仕組み
    - 連休でなくても回復できる「分散休暇」設計
    - 繁忙期前に先に休みを押さえる逆算運用
    - 休めない人を責めず、休ませられない業務構造を点検するレビュー会
    - 新任管理職向けの「有給を止めない面談テンプレ」
    - 休暇取得を“権利”だけでなく“業務継続性の訓練”として扱う社内メッセージ設計
    - 取得率でなく「失効予定日数」を早めに見せるUI
    - 半休を取りやすくする午後会議削減日
    - 有給取得日を「隠す」のではなくチームの前提にする見せ方
    - 未取得者を個人責任で終わらせないエスカレーション基準

    ## コンテンツ設計ルール

    ### テキストの質
    - 人事企画担当がそのまま会議に持ち込める文にする
    - 抽象論で逃げない
    - ただし文量が多すぎて読みにくくならないよう整理する
    - 箇条書きに逃げすぎず、読みやすい塊にする
    - 日本語は自然で、硬すぎず軽すぎない業務文体

    ### 表現ルール
    - 「すごい」「画期的」などの安い煽りは禁止
    - 根拠なき断定は禁止
    - 効果は「効きやすい」「〜が期待できる」など自然な表現にする
    - 現場 / 管理職 / 人事 / 経営のどこが主担当かを明確にする

Firebase Hosting 要件
-------------------

### 必須

* SPA として動くこと

* 直接 `/ideas/:slug` にアクセスしても 404 にならないこと

* `firebase.json` を適切に設定すること

* build 出力先は `dist` を前提にすること

* Firebase CLI でそのままデプロイできる状態にすること

### 想定コマンド

* 開発: `npm run dev`

* build: `npm run build`

* preview: `npm run preview`

* Hosting deploy: `firebase deploy --only hosting`

### 望ましい package scripts

    {
      "dev": "vite",
      "build": "vite build",
      "preview": "vite preview",
      "deploy:hosting": "firebase deploy --only hosting"
    }

### firebase.json 要件

* Hosting の `public` は `dist`

* SPA rewrite を設定する

* 不要なキャッシュ事故を避けるため、最低限の設定を入れる

* 過剰に複雑にしない
